## 1. 危险拦截 —— "清一下测试数据"

**用户输入：** "清一下测试数据"

```markdown
## 润色后的提示词

清理数据库中的测试数据 —— 使用 TRUNCATE 而非 DROP；保留种子数据（用户角色、
系统配置）；仅清理交易相关的表；用事务包裹操作，支持回滚。

约束：
  - 禁止：使用 DROP TABLE
  - 必须：保留 migration 中定义的种子数据
步骤：
  1. 区分哪些表是测试数据、哪些是种子数据
  2. 仅对测试表生成 TRUNCATE 语句
  3. 在事务中执行
输出：被清理的表名及行数

### 推导过程

推断：
  - "清" → DELETE 或 TRUNCATE，而非 DROP TABLE（DROP 不可逆）
  - 项目使用 PostgreSQL 且 migration 里有种子数据 → 必须保留
约束：
  - 破坏性操作 → 加了事务保护和操作范围限制
```

## 2. 一句话变安全方案 —— "加个登录"

**用户输入：** "给项目加个登录"

```markdown
## 润色后的提示词

为项目添加登录认证功能 —— 实现基于 JWT 的认证，包含 token 过期策略、密码
哈希（bcrypt）、CSRF 防护和会话管理。*（待确认：认证方式 —— 见歧义）*

约束：
  - 禁止：自己实现加密算法
  - Token 不得出现在 URL 中
  - 密码不得明文存储
步骤：
  1. 确认项目技术栈和现有认证相关依赖
  2. 选择认证方式（需用户确认）
  3. 实现注册 + 登录 + token 刷新流程
  4. 添加 CSRF 防护和登录接口限流
输出：可用的认证流程、安全配置、边界情况覆盖

### 推导过程

具体化："登录" → 完整的认证系统，包含 token 策略、密码哈希标准、
CSRF 防护、会话生命周期管理
推断：
  - 现代项目最可能需要 JWT 方案（也可能是 OAuth 或 session —— 标为歧义）
  - "自己实现加密"是 AI 编码的常见陷阱 → 加入明确禁止项
歧义：
  - JWT vs OAuth vs Session → 不同架构方向 → 实现前需确认
```

## 3. "又"字暴露调试方向 —— "又挂了"

**用户输入：** "导出又挂了"

```markdown
## 润色后的提示词

修复回归的导出功能 —— 复现故障，从最近的 git 提交中定位根因，修复并添加
回归测试。

角色：调试工程师，关注根因定位和回归防护
约束：
  - 修复必须包含回归测试，防止再次出现
步骤：
  1. 检查最近的 git 提交中导出相关的改动
  2. 确认是哪个导出功能、具体报错信息
  3. 复现故障，定位根因
  4. 修复并添加回归测试
输出：根因说明、修复方案、新增的回归测试

### 推导过程

纠正：
  - "又挂了" → "回归" —— "又"说明之前修过，最近的 git 历史是第一个排查目标
推断：
  - "导出" → 数据导出（CSV/Excel 下载），而非代码模块的 export
  - "挂了" → 运行时异常或输出错误（具体表现待确认）
```

## 4. 防止 AI 越改越多 —— "重构一下"

**用户输入：** "重构一下这个函数"

```markdown
## 润色后的提示词

重构此函数 —— 优化内部结构，同时保持外部行为和所有现有测试不变。
*（待确认：重构方向 —— 见歧义）*

约束：
  - 禁止：改变外部 API 签名
  - 禁止：修改现有测试用例
  - 禁止：动不相关的文件
步骤：
  1. 分析函数的职责
  2. 识别可拆分或提取的部分
  3. 逐步重构，每步后验证测试
输出：重构后的代码、变更摘要、测试结果

### 推导过程

具体化："重构" —— 改变内部结构而不改变外部行为。补充三条禁令：
不改 API 签名、不动测试、不碰无关文件。
歧义：
  - 拆分为小函数 vs 提取接口 vs 重写实现 → 不同方向 → 动手前确认
```

## 5. 缩写连击 —— "修一下 npe"

**用户输入：** "OrderSvc 里 calcTotal 有个 npe 修一下"

```markdown
## 润色后的提示词

修复 OrderService.calculateTotal() 中的 NullPointerException —— 定位
空指针访问点，添加防御性空值检查，验证修复。

### 推导过程

纠正：
  - "npe" → NullPointerException（Java 标准缩写）
  - "OrderSvc" → OrderService（标准类命名规范）
  - "calcTotal" → calculateTotal()（方法名缩写展开）
推断：
  - 计算方法中的 NullPointerException → 大概率是对可空对象的未检查字段
    访问（如 item.getPrice()，而 item 为 null）
```

## 6. 不只是写代码 —— "整理会议纪要"

**用户输入：** "整理一下会议纪要"

```markdown
## 润色后的提示词

将会议笔记整理为结构化记录。

角色：项目跟进人，关注责任落实和后续追踪
步骤：
  1. 提取会议主题和日期
  2. 列出关键决策和结论
  3. 生成待办事项表：任务 / 负责人 / 截止日期
约束：
  - 禁止：添加笔记中没有的决策
  - 禁止：将不同议题合并
  - 必须：保留每条决策的原始语境，避免断章取义
输出：主题 / 决策 / 待办事项（含负责人和截止日期）/ 关键分歧点

### 推导过程

具体化："整理" → 按「主题 / 决策 / 待办事项 / 关键分歧点」结构化输出，
附带明确的负责人和截止日期 —— 不是一段随意的总结
```

## 7. 不知道怎么回的邮件 —— "帮我回一下"

**用户输入：** "帮我回一下这封邮件"

```markdown
## 润色后的提示词

起草这封邮件的回复 —— 根据发件人身份和邮件语境，保持合适的语气，
逐一回应邮件中的要点。*（待确认：关系和语气 —— 见歧义）*

约束：
  - 禁止：未经用户确认就承诺截止日期或交付物
步骤：
  1. 确认发件人身份、关系和语境
  2. 列出需要回应的要点
  3. 按合适的语气起草回复
输出：回复草稿、回应要点清单、需要确认的承诺项

### 推导过程

推断：
  - "回复"意味着邮件期望得到具体回应
  - 语气取决于发件人身份 —— 标为歧义
歧义：
  - 客户（正式）vs 同事（随意）vs 上级（恭敬）→ 不同语气 → 确认
```

## 8. 模糊反馈变可执行 —— "精简一下"

**用户输入：** "这篇太长了，精简一下"

```markdown
## 润色后的提示词

审查文稿的精简空间 —— 找出冗余段落、收紧啰嗦的句子，在保留核心论点
的前提下提出精简版本。*（待确认：精简幅度 —— 见歧义）*

约束：
  - 必须：保留核心论点和结论
  - 禁止：直接删除用户未标注的内容 —— 只提出删减建议并说明理由
步骤：
  1. 统计当前字数，提出目标字数
  2. 找出重复或低价值的段落
  3. 收紧啰嗦的句子
  4. 展示修改前后对比及删减理由
输出：精简后的文稿、删减清单及理由、前后字数对比

### 推导过程

具体化："精简" → 具体操作（去冗余、收紧句子），附可衡量的目标（字数缩减）
歧义：
  - 删 10% vs 砍掉一半 → 完全不同 → 确认目标字数
```
